# 网上书店系统软件需求规格说明书

## 一、文档说明

本文档是在实验七初稿基础上，结合实验八的动态建模练习进一步完善后的软件需求规格说明书。

本次完善的重点是把系统中的动态行为、状态变化、业务规则和数据流动补充到需求文档中，使其更适合作为后续设计、实现和测试的依据。

## 二、范围

### 1. 项目标识

- 项目名称：网上书店系统
- 文档名称：软件需求规格说明书

### 2. 系统概述

网上书店系统是一个面向课程设计的 Java Web 应用，主要实现图书浏览、用户注册登录、购物下单和后台管理等功能。

本系统既包含前台面向用户的业务，也包含后台面向管理员的维护功能。当前文档不仅描述静态结构，也开始补充动态行为。

### 3. 文档用途

本说明书主要用于：

- 统一项目当前阶段的需求理解
- 支撑后续详细设计和编码实现
- 为测试和验收提供业务规则基础

## 三、总体描述

### 1. 产品透视

系统采用典型 B/S 架构，主要由以下几层组成：

- 表现层：JSP、HTML、CSS、JavaScript
- 控制层：Servlet
- 数据访问层：DAO
- 数据层：MySQL

### 2. 用户角色

| 角色     | 描述                                             |
| -------- | ------------------------------------------------ |
| 游客     | 浏览图书、搜索图书、查看详情、查看网站内容、注册 |
| 注册用户 | 登录、加入购物车、提交订单、查看订单、确认收货   |
| 管理员   | 登录后台，维护图书、分类、用户、订单及网站内容   |

### 3. 运行环境

- 操作系统：Windows
- JDK：课程环境要求版本
- Web 容器：Tomcat
- 数据库：MySQL
- 浏览器：Chrome、Edge 等现代浏览器

### 4. 约束条件

- 本项目不包含真实在线支付
- 当前需求仍可随着课程推进继续修订
- 项目采用传统 Java Web 单体结构实现

## 四、功能需求

### FR-01 首页展示

系统应支持首页展示推荐图书、新书上架以及网站基础信息入口。

### FR-02 图书分类浏览

系统应支持按分类浏览图书列表，并支持分页显示。

### FR-03 图书搜索

系统应支持按关键字搜索图书，并返回匹配结果列表。

### FR-04 图书详情查看

系统应支持查看单本图书详情，包括：

- 书名
- 作者
- 出版社
- 价格
- 简介
- 图片
- 分类信息

### FR-05 用户注册

系统应支持新用户注册。

要求：

- 用户名唯一
- 注册信息应完整
- 注册成功后可用于登录

### FR-06 用户登录与退出

系统应支持普通用户登录、保持会话状态以及退出登录。

### FR-07 验证码校验

系统应支持登录或关键操作中的验证码校验。

### FR-08 购物车管理

系统应支持购物车功能，包括：

- 加入购物车
- 修改商品数量
- 删除购物项
- 清空购物车

### FR-09 提交订单

系统应支持用户将购物车内容生成订单。

前置条件：

- 用户已登录
- 购物车中存在商品

后置结果：

- 系统生成订单记录
- 系统生成订单项记录
- 购物车被清空

### FR-10 查看订单

系统应支持用户查看自己的订单列表和订单详情。

### FR-11 确认收货

系统应支持用户对已发货订单执行确认收货。

### FR-12 管理员登录

系统应支持管理员登录后台。

### FR-13 用户管理

系统应支持管理员查看、查询、修改和删除普通用户。

### FR-14 分类管理

系统应支持管理员维护图书分类。

### FR-15 图书管理

系统应支持管理员维护图书信息，包括新增、修改、删除、查询和详情查看。

### FR-16 订单管理

系统应支持管理员管理订单，包括：

- 查看订单列表
- 查看订单详情
- 发货处理
- 删除订单

### FR-17 网站内容维护

系统应支持管理员维护：

- 网站简介
- 公告
- 留言
- 留言回复和删除

## 五、动态行为需求

本章是在实验八中重点补充的内容，用于描述系统中的状态变化与流程约束。

### 1. 订单状态需求

系统中的订单状态至少包括：

- 状态 1：待处理
- 状态 2：已发货
- 状态 3：已完成

状态流转要求如下：

1. 新生成订单的初始状态必须为“待处理”。
2. 管理员可以将“待处理”订单改为“已发货”。
3. 用户只能对“已发货”订单执行确认收货。
4. 用户确认收货后，订单状态变为“已完成”。
5. 不允许直接从“待处理”跳转到“已完成”。

### 2. 下单流程需求

用户下单流程应满足以下逻辑：

1. 用户已登录。
2. 购物车中存在商品。
3. 用户发起下单请求。
4. 系统生成订单主记录。
5. 系统为购物车中的每项商品生成订单项。
6. 系统计算订单总金额。
7. 系统清空购物车。

### 3. 权限行为需求

- 游客不能直接提交订单
- 游客不能查看个人订单
- 管理员不能替普通用户执行确认收货
- 普通用户不能访问后台管理功能

## 七、数据需求

### 1. 核心实体

当前系统的核心实体包括：

- 管理员
- 用户
- 图书
- 图书分类
- 图书图片
- 订单
- 订单项
- 网站信息
- 公告
- 留言

### 2. 核心关系

- 一个用户对应多个订单
- 一个订单包含多个订单项
- 一个订单项对应一本图书
- 一本图书属于一个分类
- 一本图书关联一张图片

### 3. 数据规则

- 用户名不能重复
- 购物车中商品数量必须大于 0
- 订单金额应等于订单项金额之和
- 图书必须属于某个分类
- 图书应关联图片信息

## 八、逻辑约束需求

结合实验八中 OCL 逻辑约束的思路，可将当前关键规则整理如下：

1. `购物车项数量 > 0`
2. `订单金额 = 所有订单项金额总和`
3. `未发货订单不能确认收货`
4. `用户只能查看自己的订单`
5. `图书新增时必须选择分类`
6. `图书新增时必须关联图片`

这些规则虽然当前以自然语言表达，但后续可以继续细化为更严格的逻辑约束。

## 九、非功能需求

### 1. 可用性

- 页面结构清晰
- 关键操作有明确提示
- 用户较容易理解主要业务流程

### 2. 性能

- 常见查询功能应在可接受时间内完成
- 图书列表与订单列表应支持分页

### 3. 安全性

- 后台应限制未登录访问
- 用户不能访问他人订单
- 非法输入不应导致系统异常退出

### 4. 可维护性

- 文档、代码和模型需同步更新
- 模块职责应尽量清晰
- 命名方式保持一致

### 5. 可测试性

- 功能不仅可通过页面验证，也可按状态变化和约束进行验证

## 十、动态建模对应说明

本次实验中，SRS 与动态建模之间的关系主要体现在：

### 1. 状态图层面

状态图帮助明确订单状态如何变化，以及哪些跳转是合法的。

### 2. Petri 网层面

Petri 网帮助明确订单流程中的状态和变迁关系，适合说明顺序性和前置条件。

### 3. 数据流图层面

数据流图帮助明确用户输入、系统处理、数据库写入和结果输出之间的关系。

### 4. OCL 约束层面

OCL 思路帮助把系统中的业务规则整理成更清晰、可验证的约束表达。

## 十一、验收要求

当前阶段可通过以下方式验证需求：

- 演示主要业务流程
- 检查订单状态能否正确流转
- 检查非法操作是否能被限制
- 检查主要数据对象关系是否与需求一致
