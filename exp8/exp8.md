## 二、实验目的

1. 学习 Petri 网的基本知识，理解其在系统建模中的作用。
2. 练习使用状态图、Petri 网、数据流图、OCL 逻辑等动态建模方式分析项目需求。
3. 将动态建模结果补充到当前网上书店项目的需求规格中。
4. 继续完善小组项目的 SRS，使其不仅能描述“系统有什么”，还能描述“系统如何变化”。

## 三、实验内容

### 1. 学习 Petri 网基础

本次实验首先阅读 Petri 网相关材料，目的是理解：

- 什么是库所和变迁
- 什么是标识
- Petri 网为什么适合描述状态变化和并发关系
- 在课程项目里可以用它分析哪些业务过程

### 2. 动态建模练习

结合网上书店项目当前需求，对不同业务场景尝试进行动态建模，主要包括：

- 状态图
- Petri 网
- 数据流图
- OCL 逻辑约束

### 3. 用动态模型补充 SRS

在实验七 SRS 初稿的基础上，把动态行为、状态变化和业务规则进一步写清楚，使文档更完整。

## 四、Petri 网学习情况

### 1. 对 Petri 网的基本理解

通过学习后我们认为，Petri 网特别适合描述“状态如何变化”以及“多个条件是否同时满足”的问题。

和单纯的文字描述相比，Petri 网更容易帮助大家看清楚：

- 某个业务流程是否能正常走通
- 哪些状态是允许到达的
- 哪些操作必须按顺序发生

### 2. 为什么本项目适合用 Petri 网分析

对于网上书店项目来说，最适合用 Petri 网分析的是订单相关流程。因为订单从创建到完成，本身就带有明显的状态变化和顺序约束，例如：

- 用户先有购物车
- 再提交订单
- 管理员再发货
- 用户最后确认收货

这个过程如果只用一句话描述，很容易忽略一些前置条件；而 Petri 网可以帮助把这些逻辑关系表达得更清楚。

## 五、动态建模练习情况

### 1. 状态图思路

本次实验中重点分析了订单状态的变化过程。

当前项目中的订单状态可以抽象为：

- 状态 1：待处理
- 状态 2：已发货
- 状态 3：已完成

由此可以得出一个比较清晰的状态变化路径：

`待处理 -> 已发货 -> 已完成`

同时，小组也进一步明确了几个限制条件：

- 只有管理员可以执行“发货”
- 只有已发货订单才能被用户确认收货
- 用户不能跳过“已发货”直接把订单改为“已完成”

这部分内容有助于把原来“订单管理”这种笼统功能写得更细。

### 2. Petri 网思路

在 Petri 网角度下将订单流程理解为一组状态和变迁：

- 购物车已准备好
- 用户提交订单
- 系统生成订单
- 管理员发货
- 用户确认收货

通过这种方式，我们进一步认识到：

- 订单流程中的每一步都不是独立的
- 某些操作必须以前一个状态已经满足为前提
- 系统中所谓“业务规则”，很多本质上就是状态变化约束

### 3. 数据流图思路

本次实验也继续从数据流角度梳理了系统中几个主要对象之间的关系。

动态视角下，更应该关注的是：

- 用户输入了什么数据
- 系统处理后输出什么结果
- 数据最终写到哪些表中

例如在下单场景中：

- 用户输入：购物车内容、收货信息、下单请求
- 系统处理：生成订单、生成订单项、更新购物车状态
- 系统输出：订单结果、订单状态、订单详情

这样做的好处是能让“功能需求”和“数据变化”之间的联系更加清楚。

### 4. OCL 逻辑约束思路

虽然本次实验没有把 OCL 写成非常严格的形式化表达，但通过 OCL 思路整理出了若干“必须满足的规则”，例如：

- 购物车中的商品数量必须大于 0
- 订单金额必须等于订单项金额总和
- 用户只能查看自己的订单
- 图书必须属于某个分类
- 图书应当关联图片信息

## 六、动态建模对 SRS 的补充作用

通过本次实验，对 SRS 的理解又进一步深化了。实验七更多解决的是“结构和对象”，实验八则开始处理“状态和变化”。

### 1. 功能描述更完整

以前文档中可能只写“系统支持提交订单”，但经过动态建模后，需要进一步说明：

- 什么情况下可以提交
- 提交后会发生什么
- 哪些数据会变化
- 后续还能做哪些操作

### 2. 业务规则更明确

动态建模帮助小组发现，很多问题其实不是“有没有功能”，而是“功能能不能按正确顺序执行”。

例如：

- 空购物车不能下单
- 未发货订单不能确认收货
- 后台发货只能作用于待处理订单

这些规则如果不写进 SRS，后续开发和测试就很容易理解不一致。

### 3. 更方便后续测试

有了状态变化和约束描述后，后续测试不再只是“点一下页面看看能不能用”，而是可以更明确地检查：

- 状态是否正确流转
- 非法操作有没有被拦截
- 数据是否按预期变化

## 七、本次对项目需求的进一步明确

通过动态建模，本周进一步明确了以下需求点：

1. 订单状态必须遵循固定顺序流转。
2. 用户和管理员在订单流程中拥有不同的操作权限。
3. 购物车、订单和订单项之间的关系要在需求中明确体现出来。
4. 部分页面操作背后其实对应的是状态变化，而不只是简单的按钮点击。
