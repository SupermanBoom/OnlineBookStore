## 四、过程模型讨论情况

### 1. 瀑布模型

瀑布模型的优点是阶段划分清楚，需求、设计、编码、测试等过程比较规范，适合需求稳定、文档要求明确的项目。

但是对于本组的网上书店项目来说，前期需求还在不断细化，很多功能是在实验推进中慢慢明确的。如果一开始就完全按瀑布模型执行，后期修改成本会比较高，文档和实现也容易脱节。

### 2. 原型模型

原型模型适合前期需求不够清晰的情况，可以先做一个初步页面或功能原型，再根据反馈调整。

小组认为，网上书店项目在前台页面、功能展示和管理员使用流程上，确实适合先做出基本原型，再逐步补充细节。因此原型思想对本项目是有帮助的。

### 3. 增量/迭代模型

增量模型适合把系统拆成若干部分，按模块逐步完成，比如先做登录注册，再做图书浏览，再做购物车和订单等。

对于当前项目，小组认为这种方式比较适合，因为网上书店本身就可以拆分成多个相对独立的模块，分阶段完成更容易控制进度，也更适合组内分工。

### 4. 敏捷开发

敏捷开发强调持续沟通、快速反馈、逐步迭代，适合需求会变化、团队需要频繁协作的项目。

小组讨论后认为，课程项目虽然规模不大，但需求会随着实验内容推进不断变化，因此单纯用传统模型不太灵活。敏捷思想更符合本组当前状态，尤其适合边做边改、边讨论边调整的开发过程。

## 五、小组对项目过程模型的选择

综合比较后，小组认为网上书店项目不适合完全照搬单一模型，而更适合采用“增量开发 + 敏捷推进”的方式。

具体理解如下：

- 在总体安排上，先大致确定需求、数据库和模块划分。
- 在实现过程中，以模块为单位逐步推进，比如先做用户模块，再做图书模块，再做订单和后台管理模块。
- 每完成一部分，就及时回头修改文档和补充说明。
- 如果在实验过程中发现需求变化，则优先调整文档和分工，再继续实现。

这种方式比纯瀑布模型更灵活，也比完全无计划地开发更有条理，适合大学生课设项目。

## 六、Scrum 学习情况

通过阅读 Scrum 相关资料，小组对 Scrum 的理解主要包括以下几点：

### 1. Scrum 的核心角色

- Product Owner：负责整理需求、明确优先级
- Scrum Master：负责推进过程、协调团队
- Development Team：负责具体实现工作

结合本组项目，虽然没有完全严格区分角色，但可以借鉴 Scrum 的思路：由部分成员优先整理需求和文档，其余成员负责具体模块实现，大家再定期同步进度。

### 2. Scrum 的核心事件

Scrum 强调短周期迭代和频繁沟通，一般包括：

- 迭代计划
- 日常沟通
- 迭代评审
- 迭代回顾

对于课程项目来说，小组不一定严格按企业开发方式执行，但可以把每周实验看作一个小迭代周期，在每周结束时完成一次阶段总结和调整。

### 3. Scrum 的核心思想

Scrum 最重要的不是名词本身，而是以下几种工作方式：

- 任务拆分
- 优先级管理
- 每周持续沟通
- 先完成可用部分，再逐步完善
- 及时根据问题做调整

小组认为，这种方法很适合当前项目推进，因为每次实验都会带来新的要求或新的文档任务，使用 Scrum 思想有助于减少混乱。

## 七、项目风险分析与分级

结合当前项目阶段，小组对网上书店项目的风险做了进一步细化。

| 风险编号 | 风险内容                           | 概率 | 影响 | 风险等级 | 应对措施                   |
| -------- | ---------------------------------- | ---- | ---- | -------- | -------------------------- |
| R1       | 需求继续变化，导致前后文档不一致   | 高   | 高   | 高       | 每次实验后同步更新文档     |
| R2       | 组员对 Java Web 项目结构理解不一致 | 中   | 高   | 高       | 统一梳理项目结构和模块职责 |
| R3       | 前后台功能点较多，后期容易返工     | 中   | 高   | 高       | 优先明确核心功能范围       |
| R4       | 小组协作不统一，代码和文档分散     | 高   | 中   | 高       | 使用 GitHub 统一保存资料   |
| R5       | 数据库设计不完善，后期修改成本高   | 中   | 中   | 中       | 尽早梳理主要数据表和关系   |
| R6       | 页面、编码、部署环境出现兼容问题   | 中   | 中   | 中       | 统一编码和运行环境         |
| R7       | 测试不足，后期集成时暴露较多问题   | 中   | 中   | 中       | 每完成一部分就做基础验证   |
| R8       | 实验周期有限，后期文档补写压力大   | 高   | 中   | 高       | 实验过程中同步整理文档     |

### 风险分级说明

- 高风险：需要优先关注，必须提前有预案
- 中风险：需要持续跟踪，出现问题后及时处理
- 低风险：目前影响较小，可在后续视情况处理

### 当前最需要关注的风险

现阶段小组认为最需要关注的是以下三项：

1. 需求变化风险
2. 协作方式不统一风险
3. 文档和代码不同步风险

因为这三类问题一旦放到后面再处理，容易影响整个项目推进效率。
