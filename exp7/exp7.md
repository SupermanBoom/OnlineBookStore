# 实验七 软件需求规格说明 SRS（2）

## 一、实验信息

- 课程名称：软件工程
- 实验名称：实验七 软件需求规格说明 SRS（2）
- 实验时间：2026.5.8
- 项目名称：网上书店管理系统
- 小组成员：王科杰、宁思宇、邱志远、王跃如、张凯

## 二、实验目的

1. 在实验六已有需求整理的基础上，进一步形成自己项目的 SRS 初稿。
2. 学习较新的 SRS 写法，理解国标模板和《掌握需求过程》中的需求文档思路有什么差异。
3. 通过静态建模方式帮助梳理系统结构，与用户和组员进行沟通。
4. 用建模结果补充完善当前阶段的需求规格。

## 三、实验内容

### 1. 搜集 SRS 编写案例

本次实验先对课程提供的 SRS 模板和已有案例进行整理，目的是先搞清楚一份 SRS 常见会写哪些部分，以及哪些部分对课程项目最有用。

### 2. 编写项目 SRS 草稿

在实验六阶段版本基础上，继续整理网上书店项目的需求规格说明，形成更接近“正式初稿”的 SRS 文档。

### 3. 对比不同 SRS 模板

阅读《掌握需求过程（第 3 版）》附录中的模板，对比课程国标模板，分析两者在写法、结构和关注重点上的不同。

### 4. 用静态建模辅助需求分析

通过 E-R、UML 等静态建模方式，把网上书店系统中已经明确的角色、对象和模块关系进一步整理出来，并用这些模型反过来检查需求文档是否完整。

## 四、SRS 编写案例学习情况

通过对案例和模板的整理，小组发现大多数 SRS 虽然格式不同，但核心内容比较稳定，通常都包括：

- 项目背景和目标
- 系统范围
- 用户角色
- 功能需求
- 数据需求
- 外部接口需求
- 非功能需求
- 验收或追踪说明

因此，小组在本次实验中没有追求把模板里的所有细项都机械补满，而是优先保证以下内容完整：

1. 系统做什么
2. 谁来使用
3. 核心功能有哪些
4. 主要数据对象有哪些
5. 后续设计和编码可以据此开展

## 五、国标 SRS 与《掌握需求过程》模板对比

本次实验中，小组对两类模板进行了简单比较。

| 对比项             | 国标 SRS 模板                    | 《掌握需求过程》附录模板       | 本组理解                                  |
| ------------------ | -------------------------------- | ------------------------------ | ----------------------------------------- |
| 结构特点           | 章节完整、格式规范               | 更强调需求沟通和角色视角       | 国标更正式，附录模板更贴近分析过程        |
| 关注重点           | 接口、环境、约束、可追踪性较完整 | 更强调用户目标、场景和业务特性 | 两者可以互补                              |
| 写作风格           | 偏正式、偏完整                   | 偏实用、偏需求沟通             | 课程项目适合结合使用                      |
| 对课程项目的适配性 | 适合提交正式文档                 | 适合前期讨论和修订             | 本组采用“国标结构 + 用户视角补充”的方式 |

### 小组结论

- 国标模板适合作为最终提交或阶段性正式文档框架
- 《掌握需求过程》更适合帮助大家从“用户到底要什么”这个角度思考需求
- 对课程项目来说，用国标结构来组织文档，再吸收后者中更贴近实际的部分

因此，本次实验对 SRS 的补充重点放在：

- 角色视角
- 典型功能场景
- 需求条目化
- 数据对象之间的关系

1. 文档说明

## 七、静态建模练习情况

### 1. 为什么要做静态建模

在本次实验中认识到，仅靠文字描述需求容易出现两个问题：

- 组员对系统结构理解不一致
- 写出来的需求条目和真实对象关系对应不上

因此，静态建模的意义主要不是“为了画图而画图”，而是帮助大家更清楚地表达系统结构。

### 2. 本次涉及的静态建模内容

本次实验中重点考虑了以下几类模型：

- 用例图
- E-R 图
- 类图

### 3. 用例图层面的理解

通过用例图思路，进一步明确了系统中的主要角色和用例：

- 游客：浏览图书、搜索图书、查看详情、注册
- 注册用户：登录、加入购物车、提交订单、查看订单、确认收货
- 管理员：管理图书、分类、订单、用户、网站内容

这帮助我们把前台功能和后台功能的边界看得更清楚。

### 4. E-R 图层面的理解

通过 E-R 图思路进一步明确了系统中的核心数据对象：

- 用户
- 图书
- 图书分类
- 订单
- 订单项
- 留言
- 公告
- 网站信息

并且明确了几个关键关系：

- 用户与订单是一对多
- 订单与订单项是一对多
- 图书与分类相关联
- 图书与图片相关联

### 5. 类图层面的理解

通过类图思路，小组进一步认识到当前项目的结构是：

- 表现层：JSP
- 控制层：Servlet
- 数据访问层：DAO
- 实体对象层：Book、User、Order、Catalog 等

这使得文档里“系统总体结构”和“功能需求”之间的关系更清楚了。

## 八、静态建模对需求整理的帮助

通过这次静态建模，小组发现需求文档能得到以下改进：

1. 更容易发现遗漏的角色或模块
2. 更容易看出数据表和功能之间的对应关系
3. 更方便和组员、老师沟通系统结构
4. 后续在设计文档中也可以继续沿用这些模型

因此，本次实验的重点并不只是“写一份 SRS”，而是让 SRS 和模型之间开始建立联系。

## 十一、实验总结

实验七让小组更清楚地认识到，SRS 不是简单把功能一条条写下来，而是要让文字需求、角色理解和系统结构能够对应起来。  通过模板对比，大家也意识到，正式文档和实际需求沟通并不是对立的。真正有用的做法，是既保留规范结构，又吸收更贴近用户和项目实际的表达方式。同时，这次静态建模练习也帮助小组发现，很多需求如果只靠文字描述，很容易写散；而一旦放到角色、对象和关系中去看，就更容易发现问题并补全内容。

因此，本次实验的最大收获是：让 SRS 从“阶段性需求整理”进一步走向“可以支撑后续设计和实现的初稿文档”。
