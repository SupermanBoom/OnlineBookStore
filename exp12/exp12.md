## 三、项目背景与本次评估对象

网上书店系统基于 Java Web 技术栈实现，主要采用 JSP + Servlet + DAO + MySQL 的分层结构。经过前期实验，项目已经具备用户注册登录、图书浏览、购物车、下单、后台管理、公告留言和统计报表等功能。

本次实验重点关注以下几类代码对象：

1. 控制层：`servlet/book`、`servlet/admin` 下的各类 `Servlet`
2. 数据访问层：`dao` 与 `dao/impl`
3. 公共支撑层：`util`、`filter`、`listener`

其中，`OrderSubServlet` 被选为本次 DI 分析和示例改造对象，因为它既有明确的业务价值，又存在较典型的“控制层直接创建多个 DAO 实现对象”的问题，适合说明设计原则。

## 四、依赖注入（DI）学习与理解

### 4.1 DI 的基本思想

依赖注入的核心思想不是“使用某个特定框架”，而是把“对象需要什么依赖”与“这些依赖如何创建”分离开来。传统写法常常在类内部直接 `new` 出具体实现，这会导致：

1. 类直接依赖具体实现，而不是抽象接口。
2. 类既负责业务处理，又负责依赖创建，职责变重。
3. 替换实现、编写测试桩、做局部重构时成本较高。

DI 的目标是：让对象只声明自己需要哪些依赖，而依赖对象由外部传入。常见方式包括：

1. 构造注入：通过构造器传入依赖，适合必需依赖。
2. Setter 注入：通过 setter 方法设置依赖，适合可选依赖。
3. 接口注入：通过特定接口暴露注入行为，但在现代 Java Web 项目中较少单独使用。

### 4.2 DI 与依赖倒转原则的关系

依赖注入是实现依赖倒转原则的一种常见技术手段。依赖倒转原则强调：

1. 高层模块不应该依赖低层模块，二者都应该依赖抽象。
2. 抽象不应该依赖细节，细节应该依赖抽象。

在本项目中，`Servlet` 作为较高层的控制组件，如果在类内部直接写 `new OrderDaoImpl()`、`new BookDaoImpl()`，就等于高层模块直接依赖了低层细节实现。即使代码功能正确，也会提高耦合度。

## 五、项目中的设计原则分析

### 5.1 里氏替换原则

里氏替换原则要求：子类型对象必须能够替换父类型对象而不破坏程序正确性。

本项目中的积极体现：

1. `BookDaoImpl implements BookDao`
2. `OrderDaoImpl implements OrderDao`
3. `OrderItemDaoImpl implements OrderItemDao`

这说明数据访问层已经具备“面向接口使用”的基础。只要调用方依赖接口，后续就可以替换不同实现。

当前问题：

1. 虽然定义了接口，但不少 `Servlet` 仍然直接 `new XxxDaoImpl()`。
2. 接口的抽象价值没有完全传递到上层调用代码中。

改进建议：

1. 上层尽量以接口类型持有依赖。
2. 把具体实现的创建放到构造器、工厂或框架容器中。

### 5.2 单一职责原则

单一职责原则要求一个类只承担一类主要职责。

本项目中的较好体现：

1. `BookDaoImpl` 主要负责图书数据访问。
2. `FilterLogin` 主要负责后台登录过滤。
3. `ListenerCount` 主要负责访问统计相关监听行为。

本项目中的不足：

1. 若干 `Servlet` 同时承担参数读取、登录判断、业务编排、数据访问调用、错误提示和页面跳转。
2. `DbUtil` 把连接创建、SQL 执行、资源关闭、异常处理全部集中在一个静态工具类中，后续扩展事务或连接池时会比较吃力。

以 `OrderSubServlet` 为例，它同时负责：

1. 接收请求与动作分发
2. 检查登录与购物车状态
3. 校验库存
4. 生成订单与订单项
5. 更新库存
6. 组织页面跳转与提示信息

这使得控制层职责偏重，后续若增加事务、支付、优惠券等逻辑，会进一步膨胀。

### 5.3 开闭原则

开闭原则强调：对扩展开放，对修改关闭。

本项目中的体现：

1. DAO 层接口与实现分离，增加新的 DAO 实现存在基础条件。
2. 页面、控制、数据访问分层后，新增部分功能时可以沿现有结构扩展。

本项目中的不足：

1. `Servlet` 中较多 `switch(action)` 与直接依赖具体实现，新增行为时往往需要改动原有类。
2. 某些 SQL 拼接方式可维护性一般，例如根据字符串直接拼接 `ids` 或关键字查询，后续扩展条件组合会越来越复杂。

改进建议：

1. 逐步把复杂业务移入独立服务类。
2. 对重复查询模式提炼公共方法，减少“每加一个需求就改一段旧逻辑”的情况。

### 5.4 迪米特法则

迪米特法则强调“最少知道原则”，对象应尽量少了解其他对象的内部结构。

本项目中有一定体现，例如 JSP 主要通过请求属性展示数据，没有直接接触数据库细节。

但也存在一些典型问题：

1. `Servlet` 往往要显式知道多个 DAO 的存在，并顺序组织它们的调用。
2. 订单模块中，一个控制器既要知道订单 DAO，也要知道订单项 DAO 和图书 DAO。

这会让上层对象掌握太多协作细节。更好的做法是把“提交订单”封装为一个更高层的服务能力，由控制器只调用一个入口。

### 5.5 依赖倒转原则

依赖倒转原则是本次实验的重点。

正面基础：

1. 项目已经具备接口层，如 `BookDao`、`OrderDao`、`OrderItemDao`。

主要问题：

1. 许多 `Servlet` 虽然字段类型写成接口，但对象创建仍直接绑定 `Impl`。
2. 依赖创建逻辑散落在各个方法内部，导致复用和替换困难。

本次实验的改造示例：

`OrderSubServlet` 原先在 `orderShip`、`myOrderList`、`subOrder` 等方法内直接创建多个 DAO 实现对象。现在调整为：

1. 在类中声明 `OrderDao`、`OrderItemDao`、`BookDao` 三个接口依赖。
2. 使用默认构造器创建项目运行所需的默认实现。
3. 增加一个包级构造器，允许在测试或后续重构时从外部传入依赖对象。

这样做的收益是：

1. 把依赖创建从业务方法中移出，减少重复。
2. 让类的依赖关系更显式。
3. 为后续单元测试、模拟对象替换、引入简单工厂或 Spring 容器留下接口。

### 5.6 合成复用原则

合成复用原则强调优先使用对象组合，而不是继承。

本项目多数业务对象并没有滥用继承，整体上更偏向组合，这一点是比较健康的。比如：

1. `Order` 组合 `OrderItem` 列表与 `User`
2. `Cart` 组合多个 `CartItem`
3. `CartItem` 组合 `Book`

这种做法比为各种场景设计复杂继承层次更适合当前课程项目。

不过在控制逻辑层面，组合复用还可以继续加强，例如把库存校验、订单持久化、订单项生成封装到单独的协作对象中，而不是都放在 `Servlet` 内。

## 六、结合项目的模块评估

### 6.1 订单模块

优点：

1. 功能链路完整，已支持购物车校验、库存检查、订单生成和我的订单展示。
2. 通过 `BookDao` 的 `decreaseStock` 方法加入了库存约束，避免简单超卖。
3. DAO 接口已经存在，具备进一步重构基础。

问题：

1. `OrderSubServlet` 业务职责较重。
2. 控制层直接依赖多个 DAO 细节。
3. 缺少显式事务边界，`订单新增成功但订单项或扣库存失败` 时理论上存在一致性风险。

解决思路：

1. 先用 DI 减少方法内部 `new` 依赖。
2. 后续增加 `OrderService` 层统一处理“下单”事务。
3. 再逐步把库存校验、订单写入、订单项写入聚合为一个业务服务。

### 6.2 图书数据访问模块

优点：

1. `BookDao` 接口较完整，已覆盖查询、分页、库存更新和低库存统计等需求。
2. 数据访问逻辑集中在 `BookDaoImpl` 中，避免控制层直接写 SQL。

问题：

1. 存在字符串拼接 SQL 的写法，如名称模糊查询和部分批量 ID 查询。
2. `DbUtil` 使用静态调用模式，导致 DAO 对具体数据库工具的依赖较硬。

解决思路：

1. 优先把字符串拼接查询改为参数化查询。
2. 中长期可考虑抽象数据库访问适配层，减少 DAO 对静态工具类的直接耦合。

### 6.3 公共工具模块

优点：

1. `DateUtil`、`RanUtil`、`MathUtils` 等工具类降低了重复代码。
2. `Filter` 和 `Listener` 将横切关注点进行了集中处理。

问题：

1. `DbUtil` 同时承担了过多职责。
2. 静态工具类在测试替换、异常治理和事务管理方面不够灵活。

解决思路：

1. 后续若项目继续演进，可优先对数据库访问能力做封装升级。
2. 若引入连接池或框架，应把连接获取与 SQL 执行职责适度拆分。

## 七、依赖注入示例改造说明

### 7.1 改造前

`OrderSubServlet` 在多个方法内部直接出现：

```java
OrderDao orderDao = new OrderDaoImpl();
OrderItemDao oItemDao = new OrderItemDaoImpl();
BookDao bookDao = new BookDaoImpl();
```

这意味着：

1. 每个业务方法都耦合到了具体实现类。
2. 类的依赖分散在方法内部，不易统一观察。
3. 若后续要做测试替身、缓存代理或事务包装，修改点会较多。

### 7.2 改造后

本次改造将依赖提到类级别：

```java
private final OrderDao orderDao;
private final OrderItemDao orderItemDao;
private final BookDao bookDao;
```

并通过构造器注入：

```java
public OrderSubServlet() {
    this(new OrderDaoImpl(), new OrderItemDaoImpl(), new BookDaoImpl());
}
```

同时提供包级构造器用于外部传入依赖。

### 7.3 改造效果评价

1. 没有改变系统原有业务流程和页面行为。
2. 代码更符合“依赖抽象而非依赖细节”的方向。
3. 这是一个小范围、低风险、可渐进复制到其他 `Servlet` 的改造方式。
4. 它仍然只是 DI 的起点，尚未解决事务边界和服务层缺失问题。


## 八、实验结论与体会

通过本次实验，可以更清楚地看到：课程项目即使规模不大，也会很自然地暴露出职责集中、控制层偏重、依赖写死等典型设计问题。设计原则的价值不在于背诵定义，而在于帮助我们发现“为什么这段代码后续会越来越难改”。

依赖注入并不一定意味着马上引入大型框架。对于当前网上书店项目，先从局部模块做构造注入、减少方法内部 `new`、强化接口依赖，就是一种务实且低风险的实践方式。它既能满足课程实验目标，也为后续继续引入 `Service` 层、统一事务和测试替身打下基础。

## 十、最终结论

1. 本项目已经具备一定的面向对象设计基础，尤其体现在 DAO 接口与实现分离、业务对象以组合为主等方面。
2. 当前最突出的问题是控制层职责过重，以及高层模块对低层实现的直接依赖。
3. 依赖注入是改进当前结构的合适切入点，本次以 `OrderSubServlet` 为例完成了最小化示范。
4. 后续若继续优化，建议优先补充 `Service` 层、事务边界和更规范的数据库访问封装。
