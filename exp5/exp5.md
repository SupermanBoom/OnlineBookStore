### 3. 人力资源组织结构讨论

通过本题可以看出，项目并不是所有活动都同样重要，真正决定总工期的是关键路径上的活动。对于本题来说，`A-D-H-L-N-O-P` 上的任何一个活动发生延迟，都会直接导致整个项目延期。

这也说明在项目管理中，关键路径分析的意义在于帮助团队把注意力优先放在影响总工期最大的任务上，而不是平均分配精力。

## 六、人力资源组织结构与工作方式讨论

### 1. 国内外软件开发团队方式的简单对比

本次讨论中，主要从管理方式、工作节奏和协作习惯三个方面进行了比较。

#### （1）国内团队常见特点

- 更强调执行效率和进度压力
- 层级和分工相对明确
- 加班现象相对常见
- 项目推进节奏通常较快

#### （2）国外团队常见特点

- 更强调过程规范和个人边界
- 更重视文档、评审和沟通机制
- 普遍更加重视工作与生活平衡
- 对团队成员自主性要求较高

这两种方式并不是绝对对立的。国内团队执行速度快，国外团队过程规范性更强，各有优缺点。

### 2. 对“996”等工作方式的看法

围绕实验中提到的 `996` 工作制，小组讨论后认为，对于课程项目或者正常软件开发来说，长期高强度加班并不是最理想的管理方式。短时间冲刺可能有效，但如果长期依赖高强度工作，会带来以下问题：

- 个人状态下降
- 沟通质量变差
- 缺陷增加
- 文档和代码质量下降

因此，更倾向于通过合理分工、明确优先级和持续跟踪来提高效率，而不是单纯依靠延长工作时间。

### 3. 从个人角度喜欢的工作方式

组内讨论后，大家比较认可的工作方式包括：

- 任务目标清楚
- 分工明确但可以沟通调整
- 工作环境安静、设备稳定
- 不要求长时间无意义加班
- 允许出现问题后及时讨论和修正

也就是说，大家更希望在“有计划、有沟通、压力适中”的环境里完成任务。

### 4. 从团队项目管理角度的理解

对当前这种课程项目来说，比较有效的管理方式应当包括：

1. 每周明确阶段目标
2. 按模块进行分工
3. 文档与代码同步更新
4. 出现问题时尽早暴露，而不是拖到后期

## 七、本次实验的阶段性结论

通过实验五，小组形成了以下几点认识：

1. 活动图和关键路径分析可以帮助我们更清楚地看到哪些任务会直接影响总工期。
2. 项目跟踪表应当持续更新，而不是临近提交时一次性补写。
3. 团队协作效率不只是由工作时长决定，更重要的是分工、沟通和过程管理。
4. 对课程项目来说，适合采用轻量化的跟踪方式，例如 GitHub + Markdown 表格，而不是一味追求复杂工具。

## 九、实验总结

实验五让小组开始更多地从“项目管理”的角度看待课程设计，而不只是关注功能实现本身。前几次实验更偏向需求、方法和可行性，这次实验则把注意力放在了进度控制、关键任务识别和团队协作方式上。

尤其是在活动图分析中，小组更直观地理解了关键路径的意义：真正影响项目总工期的不是所有任务，而是关键路径上的那些任务。这对后续做项目安排和任务优先级判断很有帮助。

另外，通过对项目跟踪工具和人力资源组织方式的讨论，小组也进一步明确了后续项目推进应当坚持“轻量跟踪、持续更新、合理分工、及时沟通”的思路。这对后面的开发、联调和文档整理都会有直接帮助。
