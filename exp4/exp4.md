## 四、XP 开发方法学习情况

### 1. 对 XP 的基本理解

XP 是一种强调快速反馈、持续改进和高频协作的开发方法。和传统开发方式相比，XP 更强调在开发过程中不断交流、不断测试、不断重构，而不是把所有工作都等到最后再集中处理。

小组在阅读后认为，XP 最重要的特点不是“开发得快”，而是通过一系列具体实践减少返工，提高代码和需求的一致性。

### 2. XP 的主要特点

结合资料整理，小组认为 XP 的核心特点主要包括：

- 小步快跑，功能逐步完成
- 强调客户沟通和快速反馈
- 代码实现与测试同步推进
- 鼓励持续重构，降低后期维护成本
- 倡导团队成员之间频繁交流

### 3. XP 的常见实践

本次实验中，小组重点了解了以下几种 XP 常见做法：

- 结对编程
- 测试先行
- 持续集成
- 简单设计
- 重构
- 小版本快速发布

## 五、XP 对本项目的启发

### 1. 适合借鉴的部分

虽然本组项目是课程设计，不会完全严格按照 XP 执行，但其中有些做法对当前网上书店项目很有帮助：

- 先完成最基本可运行的功能，再逐步补充
- 每完成一个模块就及时检查和修正
- 遇到结构混乱的代码时及时重构，而不是一直堆功能
- 组员之间要保持频繁沟通，避免各自做各自的内容

### 2. 结合网上书店项目的理解

对于当前网上书店项目来说，很多功能可以按“小步推进”的方式完成，例如：

- 先完成注册登录
- 再完成图书浏览和图书详情
- 再完善购物车和订单
- 最后逐步补后台管理和网站维护内容

这种推进方式和 XP 的思路比较接近，比一次性把所有功能都规划死更适合课程项目。

### 3. 不完全适合直接照搬的地方

XP 中有些做法在企业团队里比较常见，但在课程项目中不一定能完全做到，例如：

- 全程结对编程在小组作业中不太现实
- 测试先行需要较强的测试意识和工具基础，本组目前还比较薄弱
- 高频率小版本发布在课堂项目里也不一定每次都能严格执行

## 六、DevOps 学习情况

### 1. 对 DevOps 的基本理解

DevOps 可以理解为开发（Development）和运维（Operations）的协同。它强调的不只是写代码，还包括部署、运行、反馈、维护等一整套流程。

DevOps 的重点在于减少“开发完成了但跑不起来”或者“代码能写但不会部署”的问题，让开发和运行维护之间更加连贯。

### 2. DevOps 的主要关注点

- 开发和部署不应完全割裂
- 项目应尽量保持环境统一
- 功能完成后应尽早验证是否可运行
- 问题发现越早，修复成本越低
- 文档、代码、部署说明应尽量同步

### 3. 对课程项目的意义

虽然项目规模不大，也不是真实上线项目，但 DevOps 的一些思想对课程项目仍然有价值，例如：

- 不只是写完代码，还要确保项目能部署、能运行
- 小组成员不能只关注自己写的模块，还要关注整体联调
- 运行环境、数据库配置、编码配置等内容要提前统一

## 七、DevOps 对本项目的启发

### 1. 环境统一的重要性

当前网上书店项目使用 Java Web、Tomcat 和 MySQL，如果组员本地环境不统一，就很容易出现“有人能跑、有人跑不起来”的问题。因此，DevOps 思想提醒我们要尽早统一：

- JDK 版本
- Tomcat 版本
- MySQL 配置
- 数据库账号密码
- 项目编码方式

### 2. 部署与开发要同步考虑

课程项目里常见的问题是前期只顾着写功能，到最后部署时才发现路径、数据库、编码、过滤器等问题很多。通过本次实验，认识到部署并不是最后才考虑的事情，而应该在开发过程中同步关注。

### 3. 文档与运行说明同样重要

DevOps 不只是技术概念，也提醒我们要把运行步骤、数据库配置和部署方式写清楚。这样即使换一个人接手，也能较快把项目跑起来。

## 八、结合项目的阶段性结论

通过本次实验，小组对项目推进方式又有了进一步认识。

### 1. 在开发方法上

项目更适合采用“小步迭代、边做边改”的方式推进，这和前一次实验中得出的“增量开发 + 敏捷推进”思路是一致的。XP 进一步强化了这种理解。

### 2. 在协作方式上

项目不应只关注编码，还要同步关注联调、运行和文档更新。开发和部署检查不能完全分开做。

### 3. 在后续实践上

后续准备继续以下做法：

1. 每完成一个模块就做基础验证。
2. 文档和代码同步更新。
3. 保持统一环境和统一运行说明。
4. 优先完成核心功能，再逐步补充细节。
