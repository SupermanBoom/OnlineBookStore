# 软件工程实验报告

## 实验六 工作量估算、风险管理与 SRS 初稿

课程名称：软件工程
项目名称：网上书店系统
实验日期：2026.4.17
文档类型：实验报告
小组成员：王科杰、宁思宇、邱志远、王跃如、张凯

## 一、实验目的

1. 学习并尝试使用工作量估算方法，对课程项目进行初步规模和投入评估。
2. 分析项目推进过程中可能出现的主要风险，并形成风险分级与应对方案。
3. 学习软件需求规格说明书（SRS）的基本结构和写法，形成本项目的阶段性 SRS。
4. 建立能够反映项目进度、里程碑和成员工作量的跟踪方式，为后续实验提供依据。

## 二、实验内容

根据实验六要求，本次工作主要包括以下三个部分：

1. 结合网上书店项目的功能范围，对项目初始工作量进行估算。
2. 梳理当前项目在需求、技术、进度和协作方面的风险，建立风险管理表。
3. 依据国标软件文档结构，整理网上书店系统的软件需求规格说明书初稿。

## 三、项目概述

网上书店系统是一个基于 Java Web 的课程项目，采用 JSP + Servlet + DAO + MySQL 的典型实现方式，面向普通用户和管理员两类使用者。

当前阶段小组已经明确的核心需求包括：

1. 用户注册、登录、退出
2. 图书分类浏览、搜索、详情查看
3. 购物车与订单提交
4. 管理员对图书、分类、用户、订单和网站内容的管理

由于实验六属于项目早期阶段，本次估算和风险分析更强调“帮助控制范围”和“提前暴露问题”，而不是追求非常精确的数字。

## 四、工作量估算

### 4.1 估算方法选择

本次参考教材中的 COCOMO II 基本思路进行初步估算。考虑到课程项目规模较小、需求仍在逐步明确，因此本次估算只用于：

1. 粗略判断项目规模是否超出课程团队可承受范围
2. 帮助进行模块拆分与成员分工
3. 为后续风险管理提供依据

### 4.2 估算假设

为避免把第三方库、图片素材、样式资源全部计入，本次只估算“项目核心自编代码和需求实现工作量”。初始版本按以下范围考虑：

1. 前台基础业务：注册、登录、浏览、搜索、购物车、订单
2. 后台基础业务：图书、分类、用户、订单的基本管理
3. 数据库设计与联调
4. 阶段性文档编写与修改

按照课程项目早期原型的复杂度，估计核心规模约为 `0.8 KSLOC`。

### 4.3 COCOMO II 粗略计算

采用公式：

`PM = A × Size^E × ΠEM`

其中：

- `A = 2.94`
- `Size = 0.8 KSLOC`
- 规模因子按课程项目取中等偏低风险，得到 `E ≈ 1.07`
- 成本驱动因子综合考虑人员经验一般、工具条件普通、需求仍会调整，取 `ΠEM ≈ 1.05`

代入后：

`PM ≈ 2.94 × 0.8^1.07 × 1.05 ≈ 2.4 人月`

### 4.4 估算结果理解

以上结果说明，若按完整小型软件项目方式严格计算，当前范围已经不是“单人几天即可完成”的任务，而需要持续分工和迭代推进。

考虑课程项目的实际情况，小组对结果作如下解释：

1. `2.4 人月` 更适合作为“规模提醒”，说明项目不能无边界扩展。
2. 实际课程实施中，通过复用现有技术框架、控制范围和压缩非关键功能，可以把可交付工作压缩到课程周期内。
3. 因此，本组采用“优先保证主流程可运行，再逐步补充管理和文档”的推进策略。

### 4.5 模块工作量拆分

结合当前项目范围，对工作量进行进一步拆分：

| 模块         | 主要内容                   | 估计占比 | 说明                 |
| ------------ | -------------------------- | -------: | -------------------- |
| 用户模块     | 注册、登录、退出、基本信息 |      15% | 基础但必须优先完成   |
| 图书模块     | 分类、列表、详情、搜索     |      20% | 前台核心浏览能力     |
| 购物车与订单 | 购物车、下单、订单查看     |      25% | 业务逻辑最集中       |
| 后台管理     | 用户、图书、分类、订单管理 |      20% | 页面多、联调工作较多 |
| 数据库与工具 | 表结构、DAO、公共工具      |      10% | 为各模块提供支撑     |
| 文档与测试   | SRS、整理、验证与修订      |      10% | 课程项目必须同步推进 |

从拆分结果看，订单与后台管理是相对更重的工作区域，后续需要重点关注。

## 五、风险管理分析

### 5.1 风险识别思路

本次风险分析主要从以下几个方面展开：

1. 需求是否会变化
2. 技术栈是否熟悉
3. 多人协作是否容易产生冲突
4. 进度是否可能失控
5. 文档与代码是否可能脱节

### 5.2 风险分级表

| 风险编号 | 风险描述                                     | 概率 | 影响 | 风险等级 | 应对策略                                             |
| -------- | -------------------------------------------- | ---- | ---- | -------- | ---------------------------------------------------- |
| R1       | 需求边界不清，后续不断增加功能               | 高   | 高   | 高       | 先冻结核心主流程，新增需求必须评估是否影响当前里程碑 |
| R2       | 组员对 Java Web、Servlet、DAO 结构掌握不一致 | 中   | 高   | 高       | 统一代码结构，先完成可运行样例，再分模块开发         |
| R3       | 前后台联调时数据库字段或接口不一致           | 高   | 中   | 高       | 统一表结构说明，修改数据库前先同步组内               |
| R4       | 进度估计偏乐观，导致后期集中赶工             | 高   | 高   | 高       | 每周更新任务状态，优先完成注册、浏览、下单主链路     |
| R5       | 文档只在提交前补写，导致与代码脱节           | 中   | 中   | 中       | 每次实验结束同步更新 SRS 和说明文档                  |
| R6       | 成员分工不清，出现重复开发或遗漏             | 中   | 中   | 中       | 按模块明确负责人，并保留一名成员负责总体汇总         |
| R7       | 测试不足，功能表面可用但流程不完整           | 中   | 高   | 高       | 以用户主流程为主线进行场景化检查                     |
| R8       | 环境配置差异导致本地可运行、他人无法复现     | 中   | 中   | 中       | 统一 JDK、Tomcat、MySQL 版本和初始化脚本             |

### 5.3 当前重点风险

现阶段最需要重点关注的是以下三个风险：

1. `R1 需求膨胀`
   因为网上书店看起来功能很多，如果没有范围控制，很容易从“课程项目”变成“想做什么都加”。
2. `R4 进度偏乐观`
   实验六阶段往往容易低估后续联调、测试和文档修订的成本。
3. `R3 联调不一致`
   课程项目常见问题不是某个单独模块写不出来，而是前后台、数据库和页面之间对不上。

### 5.4 风险应对原则

本组在后续阶段采用以下风险应对原则：

1. 优先完成主业务链路，而不是平均铺开所有功能。
2. 先保证文档、数据库和代码命名一致，再追求页面和细节完善。
3. 每周至少做一次功能同步检查，避免问题积累到最后。

## 六、SRS 初稿整理情况

### 6.1 对 SRS 结构的理解

通过学习国标相关结构后，小组认为课程项目现阶段的 SRS 至少应包括：

1. 文档说明与范围
2. 总体描述
3. 功能需求
4. 外部接口需求
5. 内部数据需求
6. 环境与资源需求
7. 软件质量要求
8. 约束、验收与可追踪性说明

### 6.2 本次形成的 SRS 内容

1. 项目标识、系统概述和文档用途
2. 用户角色与运行环境
3. 前后台核心功能需求条目
4. 外部接口与主要数据对象
5. 非功能需求与基本约束
6. 初步验收与需求可追踪性说明

### 6.3 当前 SRS 的不足

由于仍处于项目较早阶段，这份 SRS 还存在一些不足：

1. 典型业务场景描述还不够细
2. 状态变化和动态规则尚未展开
3. 异常场景与边界条件写得还不够充分
4. 部分接口需求仍偏概括，后续需要继续细化

这些问题将在后续实验中通过静态建模、动态建模和设计分析继续完善。

## 八、实验结论与体会

通过实验六，小组对软件工程中“先估算、先识别风险、先整理需求”的价值有了更直观的理解。

1. 工作量估算虽然不可能完全准确，但能帮助我们判断范围是否合理。
2. 风险管理不是形式化表格，而是提前提醒我们哪些问题最容易拖慢项目。
3. SRS 不是最终阶段才需要补的文档，而是后续设计、开发、联调和测试的共同基础。

本次实验的直接成果是形成了一份可继续演进的 SRS 初稿，也让小组对后续实验的重点有了更清晰的预期：先稳住核心需求，再逐步把模型、设计和实现补齐。
