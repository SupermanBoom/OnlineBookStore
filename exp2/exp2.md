# 实验二 持续沟通修订可行性分析报告、CASE调研

## 一、实验信息

- 课程名称：软件工程
- 实验名称：实验二 持续沟通修订可行性分析报告、CASE调研
- 实验时间：2026.3.20-2026.3.26
- 项目名称：网上书店管理系统
- 小组成员：王科杰、宁思宇、邱志远、王跃如、张凯

## 二、实验目的

1. 继续对项目进行沟通和调查，在上一次实验的基础上修订可行性分析报告。
2. 调研常见 CASE 工具，了解这些工具在软件工程中的作用。
3. 结合本组项目情况，初步选择后续可能会使用的工具和协作方式。

## 三、实验内容

### 1. 修订可行性分析报告

根据本周的项目沟通结果，对上一周形成的可行性分析报告进行补充和调整，重点查看以下内容：

- 需求是否有新增或变化
- 用户角色和主要功能是否描述清楚
- 当前技术路线是否适合继续开发
- 文档内容是否能反映项目现阶段情况

### 2. 调研 CASE 工具

查找常见的 CASE 工具资料，了解它们在需求分析、系统建模、数据库设计、版本管理和测试等方面的用途，并结合课程设计项目判断其适用性。

### 3. 讨论后续协作方式

结合项目规模和组员实际能力，讨论后续开发中适合使用的协作平台、画图工具和版本管理方式，为后面的实验打基础。

## 四、本周项目沟通与修订情况

本周小组继续围绕“网上书店管理系统”进行讨论。和最开始相比，大家对项目理解更加清楚，也发现原来的可行性分析报告里有一些内容写得比较笼统，需要进一步补充。

在本次沟通中，小组主要明确了以下几点：

- 项目仍然以网上书店为主题，主要实现图书浏览、注册登录、选购和后台管理等功能。
- 除了普通用户购书功能外，还需要考虑管理员对图书、订单、用户以及网站内容的管理。
- 网站本身除了卖书功能外，还应包括简介、公告、留言等内容，这部分属于网站维护范围。
- 目前需求还处于逐步明确的过程，后续实验中仍然可能继续调整。

## 五、对可行性分析报告的主要修订

### 1. 需求方面

在原有“网上书店”主题不变的前提下，进一步明确了项目中的主要参与者：

- 普通浏览者：查看图书信息和网站内容
- 注册会员：登录后购买图书
- 管理员：维护图书、订单、用户和网站信息

相比上一次，本次修订更强调“管理员维护功能”和“网站内容维护”这两个部分，而不仅仅是前台购书流程。

### 2. 技术方面

小组认为当前选择 Java Web 方向继续完成项目是可行的，原因主要有：

- 使用 MySQL 存储数据比较方便
- Tomcat 部署方式较成熟，适合课设使用
- 这种技术方案资料多、遇到问题较容易查找解决办法


### 3. 经济和实施方面

本项目属于课程实验项目，不需要额外的商业成本，开发环境基本都可以在现有电脑环境中完成，所以经济可行性较高。

在实施方面，小组认为项目规模适中，如果分工合理、文档同步更新，是可以在课程周期内完成的。

### 4. 风险方面

通过本周讨论，也发现了一些后续需要注意的问题：

- 项目需求可能继续变化，文档要及时更新
- 部分组员对 Java Web 项目的整体结构还不够熟悉
- 前后台功能较多，如果前期规划不清楚，后面容易出现返工
- 如果没有统一的协作方式，代码和文档容易混乱

这些内容也作为风险补充进了可行性分析报告中。

## 六、CASE 工具调研情况

本周小组主要了解了几类比较常见的 CASE 工具，并做了简单比较。

| 工具 | 主要用途 | 特点 | 小组看法 |
| --- | --- | --- | --- |
| Visio | 流程图、结构图绘制 | 图形界面直观，上手快 | 适合画流程图 |
| PlantUML | UML 图绘制 | 文本方式描述，方便修改 | 适合做类图和时序图 |
| PowerDesigner | 数据库建模 | 适合做 E-R 图和表关系设计 | 可用于数据库设计 |
| Git | 版本管理 | 能记录修改过程 | 很适合小组协作 |
| GitHub | 协作平台 | 方便共享代码和文档 | 建议使用 |
| JMeter | 接口与性能测试 | 可用于后期测试 | 现阶段先了解 |
| Rational Rose / StarUML | UML 建模 | 建模功能完整 | 可作参考，不一定必须使用 |

## 七、工具分析与初步选择

### 1. 版本管理工具

小组认为 Git 和 GitHub 对课程项目非常有帮助。因为项目文档、代码和每次修改记录都可以保存下来，后续如果出现问题，也比较容易回溯。

### 2. 建模工具

在 UML 相关工具中，小组认为 PlantUML 更轻量，适合后续结合文档一起使用；Visio 适合画展示性较强的图，例如系统流程图、功能结构图等。

### 3. 数据库设计工具

PowerDesigner 适合数据库阶段使用，可以帮助理清图书、用户、订单等数据表之间的关系。虽然不一定每个成员都熟练掌握，但作为参考工具是有意义的。

### 4. 测试工具

JMeter 在本次实验中主要作为了解对象。因为当前项目还处在分析和逐步设计阶段，所以测试工具暂时不作为重点，只记录为后续可选工具。

## 八、小组讨论后的阶段结论

经过本周调研和讨论，小组形成了以下初步结论：

1. 项目仍然继续采用 Java Web 技术路线完成。
2. 可行性分析报告需要继续保留“可调整”的特点，随着需求变化不断修订。
3. 后续小组协作建议采用 GitHub 作为主要共享平台。
4. 在画图和建模方面，优先考虑 Visio、PlantUML 这类相对容易上手的工具。
5. 当前阶段重点还是把需求、功能范围和文档内容理清楚，工具选择以“够用、方便、小组能掌握”为原则。

## 十、实验收获与总结

通过这次实验，小组进一步认识到软件工程文档不是一次写完就结束，而是要随着项目推进不断更新。尤其是可行性分析报告，它不仅是前期任务，更是后续开发的重要参考。

另外，这次 CASE 工具调研也让大家对软件工程过程中的辅助工具有了更直观的认识。以前更关注代码实现，现在能够意识到流程图、建模、版本管理、协作平台这些内容同样重要。

总的来说，本次实验的重点不是把项目最终结果全部确定下来，而是围绕项目现阶段情况进行沟通、修订和调研，为下一步设计和开发做好准备。
